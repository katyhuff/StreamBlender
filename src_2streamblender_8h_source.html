11 /// @class StreamBlender
12 ///
13 /// This holds a module for cyclus that blends streams of material into a
14 /// resulting material. It aims for a goal recipe and can specify an order of
15 /// preference for using available commodities. It was designed to run as a fuel
16 /// fabrication facility, but can be used for any purpose that requires a
17 /// combining of commodities that represent material streams. It was initially
18 /// developed to run the fco code-to-code comparison.
19 ///
20 /// The StreamBlender class inherits from the Facility class and is
21 /// dynamically loaded by the Agent class when requested.
22 ///
23 /// @section introduction Introduction
24 /// The StreamBlender is a facility that receives commodities, holds onto them
25 /// for some number of months, offers them to the market of the new commodity. It
26 /// has three stocks areas which hold commods of commodities: reserves,
27 /// processing, and stocks. Incoming commodity orders are placed into reserves,
28 /// from which the processing area is populated. When a process (some number of
29 /// months spent waiting) has been completed, the commodity is converted and
30 /// moved into stocks. Requests for commodities are bid upon based on the state
31 /// of the commodities in the stocks.
32 ///
33 /// @section agentparams Agent Parameters
34 /// Place a description of the required input parameters which define the
35 /// agent implementation.
36 /// #. process_time_ : the number of timesteps a conversion process takes <0>
37 /// #. refuel_time : the number of timesteps required to reload the processing after
38 /// a process has finished <0>
39 ///
40 /// The StreamBlender also maintains a cyclus::CommodityRecipeContext, which
41 /// allows it to track incommodity-inrecipe/outcommodity-outrecipe groupings.
42 ///
43 /// @section optionalparams Optional Parameters
44 /// Place a description of the optional input parameters to define the
45 /// agent implementation.
46 ///
47 /// @section detailed Detailed Behavior
48 /// After a StreamBlender enters the simulation, it will begin requesting all
49 /// incommodities.
50 ///
51 /// As soon as it receives a commodity, that commodity is placed in the
52 /// processing storage area.
53 ///
54 /// On the tick of the timestep in which that incommodity's time is up, it is
55 /// converted to the outcommodity type, by simply changing the commodity name.
56 /// Then, it is offered to the outcommodity market.
57 ///
58 /// The StreamBlender can manage multiple input-output commodity pairs, and keeps
59 /// track of the pair that each resource belongs to. Resources move through the
60 /// system independently of their input/output commodity types, but when they
61 /// reach the stocks area, they are offered as bids depedent on their output
62 /// commodity type.
63 ///
64 /// @section requests Requests
65 /// A StreamBlender will make as many requests as it has possible input
66 /// commodities. It provides a constraint based on a total request amount
67 /// determined by its processing capacity.
68 ///
69 /// @section bids Bids
70 /// A StreamBlender will bid on any request for any of its out_commodities, as
71 /// long as there is a positive quantity of material in its stocks area
72 /// associated with that output commodity.
73 ///
74 /// @section ics Initial Conditions
75 /// A StreamBlender can be deployed with any number of commods in its reserve,
76 /// processing, and stocks buffers. Recipes and commodities for each of these
77 /// groupings must be specified.
78 ///
79 /// @todo add decommissioning behavior if material is still in stocks
80 ///
81 /// @section end End of Life
82 /// If the current time step is equivalent to the facility's lifetime, the
83 /// reactor will move all material in its processing to its stocks containers,
84 /// converted or not.
85 ///
